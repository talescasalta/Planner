# PRD — Private Couple Expense Classifier

## 1. Product Summary

This project is a private web app for classifying, reviewing, and analyzing personal and shared expenses for a couple.

The system must support three financial views:

1. Pessoa A
2. Pessoa B
3. Casal

The app should allow each person to log in separately, see only the expenses they are allowed to see, and jointly access shared couple expenses.

The app will use deterministic rules and optional LLM classification to categorize imported transactions. The LLM must only receive minimized, anonymized transaction data and must never receive full financial statements, personal identifiers, secrets, or raw files.

## 2. Primary Goal

Build a secure, maintainable SvelteKit web app that allows a couple to:

- import transactions from CSV or manual entry
- classify expenses by category
- assign expenses to Pessoa A, Pessoa B, or Casal
- distinguish who paid from who owns the expense
- review uncertain classifications
- create reusable classification rules
- view monthly reports by person, category, and shared expenses

## 3. Non-Goals

The first version should not attempt to be a full personal finance platform.

Do not implement in v1:

- bank account integrations
- Open Finance integrations
- automatic card/bank scraping
- investment tracking
- bill payment
- budget forecasting
- mobile native app
- multi-household SaaS billing
- public user signup for strangers
- complex accounting/reimbursement engine
- OCR/PDF parsing unless explicitly requested later

## 4. Tech Stack

Use:

- SvelteKit
- TypeScript
- Supabase Auth
- Supabase Postgres
- Supabase Row Level Security
- Vercel deployment
- Tailwind CSS if styling is needed
- OpenAI/OpenRouter-compatible LLM API only from server-side code

Recommended structure:

- SvelteKit for frontend and server routes/actions
- Supabase Auth for login
- Supabase Postgres for data
- RLS for hard data isolation
- Server-side LLM endpoint for classification
- Local deterministic rules before any LLM call

## 5. Core Product Concepts

### 5.1 User

An authenticated person who can access the app.

Examples:

- Pessoa A
- Pessoa B

### 5.2 Household

A shared financial space for the couple.

Initially, the app may assume one household, but the data model should support multiple households in the future if possible.

### 5.3 Financial Profile

A logical financial owner of expenses.

Required profiles:

- Pessoa A
- Pessoa B
- Casal

A financial profile is not always the same thing as an authenticated user.

### 5.4 Transaction

A financial record imported or entered manually.

A transaction must support:

- date
- description
- merchant
- amount
- currency
- source account/card, if available
- who paid
- who owns the expense
- category
- subcategory
- classification status
- review status

### 5.5 Transaction Access

Explicit access control that determines which user can read or edit a transaction.

Do not rely only on `owner_profile_id`.

Shared couple expenses must be visible to both users. Private expenses must be visible only to the allowed user.

### 5.6 Classification Rule

A reusable rule created manually or learned from confirmed classifications.

Example:

- If merchant contains `DROGASIL`, category is `Saúde / Farmácia`
- If merchant contains `PAO DE ACUCAR`, owner is usually `Casal`, category is `Mercado`

### 5.7 Classification Suggestion

A proposed classification generated by deterministic rules or LLM.

The user must be able to confirm or correct it.

## 6. Main User Stories

### Authentication

As a user, I want to log in with my own credentials so that my private expenses are not visible to my partner.

Acceptance criteria:

- unauthenticated users cannot access `/app`
- each user has a separate session
- private data is loaded server-side
- login works in production on Vercel

### Import Transactions

As a user, I want to upload a CSV of transactions so that I do not need to manually enter every expense.

Acceptance criteria:

- user can upload a CSV file
- system parses rows into normalized transactions
- user can preview imported transactions before finalizing
- duplicate detection exists at least in basic form
- raw file contents are not logged
- raw uploaded files are not committed or stored permanently unless explicitly required

### Manual Transaction Entry

As a user, I want to manually add a transaction so that I can register cash, Pix, or missing expenses.

Acceptance criteria:

- user can enter date, description, amount, who paid, owner profile, and category
- transaction visibility follows access rules
- validation prevents invalid or incomplete records

### Expense Ownership

As a user, I want to distinguish who paid from who owns the expense so that shared and individual spending are calculated correctly.

Acceptance criteria:

- `paid_by_user_id` and `owner_profile_id` are separate fields
- a transaction paid by Pessoa A can belong to Casal
- a transaction paid by Pessoa B can belong to Pessoa A
- reports can group by owner profile and by payer

### Shared Couple Expenses

As a couple, we want shared expenses to be visible to both users.

Acceptance criteria:

- Casal transactions are visible to both Pessoa A and Pessoa B
- both users can review shared transactions if they have edit access
- shared transactions are not duplicated per user

### Private Expenses

As a user, I want some transactions to remain private.

Acceptance criteria:

- Pessoa A cannot see private transactions of Pessoa B
- Pessoa B cannot see private transactions of Pessoa A
- private transaction access is enforced by database RLS, not only by frontend filtering

### Classification

As a user, I want the system to classify expenses automatically so that I spend less time organizing transactions.

Acceptance criteria:

- deterministic rules run first
- LLM is called only when rules are insufficient
- LLM receives minimized, anonymized data
- classification includes category, subcategory, owner profile, confidence, and review flag
- low-confidence classifications are marked as `needs_review`

### Review Queue

As a user, I want to review uncertain classifications so that the system becomes more accurate over time.

Acceptance criteria:

- `/app/review` shows transactions needing review
- user can confirm or correct category
- user can correct owner profile
- user can create or update classification rules based on corrections
- confirmed transactions leave the review queue

### Reports

As a user, I want to see monthly spending summaries.

Acceptance criteria:

- monthly total by Pessoa A, Pessoa B, and Casal
- spending by category
- spending by payer
- filter by month, category, profile, and review status

## 9. Access Control Requirements

Access must be enforced at the database level using Supabase RLS.

Minimum rules:

- unauthenticated users cannot read financial data
- users can only access households where they are members
- users can only read transactions where a matching `transaction_access` row grants `can_read = true`
- users can only edit transactions where `can_edit = true`
- users cannot create access records for users outside their household
- users cannot escalate their own permissions
- shared Casal transactions must have access rows for both users
- private transactions must only have access rows for the permitted user

Frontend filtering is not sufficient.

## 10. LLM Classification Requirements

The LLM is only a classifier assistant.

The LLM must not:

- decide database permissions
- read all transactions
- receive raw bank/card statements
- receive full names
- receive CPF
- receive card numbers
- receive account numbers
- receive addresses
- receive balances
- receive secrets
- execute SQL
- create users
- alter RLS policies
- permanently create rules without user review

The LLM may receive:

- transaction id
- cleaned merchant
- cleaned description
- amount
- date or month
- allowed categories
- anonymized profiles: Pessoa A, Pessoa B, Casal
- small list of relevant non-sensitive rules

The LLM should return structured JSON only.

Example expected output:

```json
{
  "category": "Saúde / Farmácia",
  "subcategory": null,
  "owner_profile": "Casal",
  "confidence": 0.84,
  "needs_review": false,
  "reason_code": "known_pharmacy_merchant"
}
